\section{Stakeholder Identification Based on IEEE 42010}

IEEE 42010 asks us to define stakeholders, their concerns and architecture views that answer those concerns.

\subsection{Stakeholders and Concerns}

\begin{longtable}{|p{3cm}|p{3cm}|p{8cm}|}
\hline
\textbf{Stakeholder} & \textbf{Role} & \textbf{Key Concerns} \\
\hline
Students & Main users for practice tests and coding challenges & Simple UI, fast evaluation under 5 seconds, correct scoring, clear analytics \\
\hline
Administrators & Manage questions and test settings & Reliable question operations, smooth test setup, usage reports \\
\hline
Developers & Build and maintain system & Clear module boundaries, testable code, easy local setup \\
\hline
University or Institution & Deployment owner & 99.5\% uptime, stable performance in placement season, data privacy, low infra cost \\

\subsection{Architecture Viewpoints and Views}

\subsubsection{Logical Viewpoint}
\begin{itemize}
    \item View: Internal module split in one modular monolith app.
    \item Main concerns: NFR6 maintainability, separation of concerns.
    \item Stakeholders: Developers and administrators.
    \item Summary: Auth, test management, evaluation and analytics are separate modules. Each module has its own routes, services and data layer. Modules interact through clear interfaces.
\end{itemize}

\subsubsection{Process Viewpoint}
\begin{itemize}
    \item View: Runtime flow for code submission to final result.
    \item Main concerns: NFR1 performance, NFR2 load handling, fault isolation.
    \item Stakeholders: Students and developers.
    \item Summary: API receives submission, stores queued state and pushes job to queue. Worker picks job, runs code in sandbox and stores final output.
\end{itemize}

\subsubsection{Deployment Viewpoint}
\begin{itemize}
    \item View: Container level deployment structure.
    \item Main concerns: NFR4 availability, NFR2 scalability, cost control.
    \item Stakeholders: University and developers.
    \item Summary: App, worker, queue and MongoDB run as separate containers. Worker talks to Docker engine for isolated runtime containers. More worker containers can be added in peak time.
\end{itemize}

\subsubsection{Security Viewpoint}
\begin{itemize}
    \item View: Authentication, authorization and code isolation.
    \item Main concerns: NFR3 security.
    \item Stakeholders: University, admins and students.
    \item Summary: JWT handles stateless auth. Role checks protect admin routes. Sandbox container enforces memory, CPU and timeout limits.
\end{itemize}

\section{Major Design Decisions (ADR Summary)}

This task has five accepted ADRs.

\subsection{ADR 001: Docker Containers for Code Execution}
\begin{itemize}
    \item Status: Accepted
    \item Why: User code is untrusted and can crash server.
    \item Decision: Run each code submission in short lived Docker container with strict limits.
    \item Effect: Strong isolation and better safety. Adds small startup delay.
\end{itemize}

Full record: \texttt{ADRs/ADR-001-Docker-Code-Execution.md}

\subsection{ADR 002: Asynchronous Message Queue}
\begin{itemize}
    \item Status: Accepted
    \item Why: Code execution takes seconds and API must stay responsive.
    \item Decision: API pushes job to queue. Worker processes it in background.
    \item Effect: Fast API response and better spike handling. Needs retry and failure handling logic.
\end{itemize}

Full record: \texttt{ADRs/ADR-002-Async-Message-Queue.md}

\subsection{ADR 003: Modular Monolith}
\begin{itemize}
    \item Status: Accepted
    \item Why: Team is small and timeline is short so microservices overhead is not practical.
    \item Decision: One deployable app with clear internal modules and strict boundaries.
    \item Effect: Easy deployment and easy local debugging. Independent module scaling is limited.
\end{itemize}

Full record: \texttt{ADRs/ADR-003-Modular-Monolith.md}

\subsection{ADR 004: JWT Stateless Authentication}
\begin{itemize}
    \item Status: Accepted
    \item Why: Need simple auth for student and admin roles.
    \item Decision: Signed JWT token with expiry and role claims.
    \item Effect: Fast auth checks and easy scaling across instances. Early token revoke is limited in prototype.
\end{itemize}

Full record: \texttt{ADRs/ADR-004-JWT-Authentication.md}

\subsection{ADR 005: MongoDB for Primary Data Storage}
\begin{itemize}
    \item Status: Accepted
    \item Why: Need to store different question types and nested execution results with fast prototype development.
    \item Decision: Use MongoDB as primary document database with Mongoose validation.
    \item Effect: Flexible schema and faster development. Data integrity checks must be handled in service logic.
\end{itemize}

Full record: \texttt{ADRs/ADR-005-MongoDB-Database-Selection.md}
