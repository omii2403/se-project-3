Idempotency

Duplicate submissions for the same test question are rejected. Idempotency key checked before enqueue.

Retry Policy

2 attempts with exponential backoff (1s base). Failed jobs move to dead-letter queue for inspection.

Design Patterns

Strategy Pattern — Evaluation Logic

Each question type has a dedicated strategy class implementing a common evaluate() interface. Adding a new type = new class, zero changes to the pipeline.

⬡
CodeStrategy
Docker exec
stdout compare
◉
McqStrategy
Case-insensitive
string match
🗄
SqlStrategy
alasql engine
+ normalized match
evaluate(submission, question) → { passed, score, output }
Factory Method Pattern — Strategy Creation

A single factory function maps question type strings to strategy instances. The processing pipeline depends on the abstraction, not concrete classes.

createEvaluationStrategy(type) { switch (type) { case 'code': return new CodeStrategy(); case 'mcq': return new McqStrategy(); case 'sql': return new SqlStrategy(); } }

✓ Single point of strategy instantiation

Architecture Decision Records

ADR-001

Docker Sandboxing for Code Execution

Untrusted code isolated in containers: 1 CPU · 256 MB · no network · 10s timeout. Trade-off: 200–500ms startup overhead per evaluation.

ADR-002

Asynchronous Queue (BullMQ + Redis)

API returns 202 in <50ms; worker evaluates asynchronously. Better burst responsiveness. Trade-off: added ops complexity (Redis + worker).

ADR-003

Modular Monolith over Microservices

5-person team, 4-week sprint — microservices overhead not justified. Strict module boundaries enforce no cross-module imports. Trade-off: coupled scaling.

ADR-004

JWT Stateless Authentication

Signed JWT with userId + role, 24h expiry. No DB lookup per request — fast and horizontally scalable. Trade-off: early revocation not supported.

ADR-005

MongoDB Document Database

Mixed question types (code/MCQ/SQL) have different fields and the schema evolved rapidly during the prototype phase. Mongoose provides validation at the application layer. Trade-off: referential integrity must be enforced in app code, not the DB.

Architectural Tactics

Performance

Async Queue Decoupling

Submission API returns 202 after DB save + enqueue (<50ms). Worker processes heavy evaluation separately. Decouples API throughput from evaluation time.

Security

Sandboxed Docker Execution

Resource limits: CPU 1 core · RAM 256 MB · no network · 10s timeout. Runaway code cannot crash the API or host system.

Performance

Cache-Aside with Invalidation

  • Submissions list: 8s TTL
  • Question topics: 60s TTL
  • Student summaries: 15s TTL
  • Max 2000 entries (LRU eviction)
  • Startup warm-up: 8 most-active students
Security

Centralised Auth + RBAC

Global middleware on /api requires valid JWT (except signup/signin). Admin routes add second role-check middleware. Safety guards prevent self-deletion and seed admin removal.

Observability

Runtime Observability Built-In

Health Probes
/health/live always 200 · /health/ready returns 503 if any dep. is down

Structured Logs
requestId · method · path · status · latency · sessionId · userId on every request

Worker Telemetry
5s heartbeat · completed/failed counters · queue depth · 5-min API latency window

Key Insights & Takeaways

🐳

Safety is Foundational

Docker sandboxing with CPU/memory/network/timeout limits is what makes untrusted code execution at scale safe. Without it, a single malicious submission could take down the server.

⚡

Async Trades Simplicity for Scale

The BullMQ queue delivers sub-500ms API response times under concurrent load, but adds operational complexity — Redis, a worker process, and dead-letter management.

🏗

Architecture Enables Teamwork

The modular monolith with strict module boundaries let 5 developers work in parallel with minimal merge conflicts — a microservices split would have been unnecessary overhead for this timeline.

🔑

Stateless Auth Scales Horizontally

JWT tokens carry role information, so no DB lookup per request. Multiple API instances can run behind a load balancer with no shared session storage required.

🔭

Observability from Day One

Health probes, structured logs, and telemetry dashboards were built into the design, not added after. This makes debugging and incident response significantly faster in production.

📐

Patterns Preserve Extensibility

Strategy + Factory patterns for evaluation keep the question-type logic isolated. Adding a new question format (e.g., drag-and-drop) requires only a new strategy class — zero changes to the pipeline.
