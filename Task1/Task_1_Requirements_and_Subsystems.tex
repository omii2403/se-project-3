\section{Functional and Non Functional Requirements}

\subsection{Functional Requirements}
\begin{itemize}
    \item FR1: The platform should allow students and admins to register, login and manage profile.
    \item FR2: Students should select topic, question type and difficulty to create custom practice test.
    \item FR3: System should create timed tests and avoid repeating same question for same user.
    \item FR4: Student code should run in isolated sandbox container so main server stays safe.
    \item FR5: After submission system should auto evaluate answers and show score in dashboard.
    \item FR6: Admin should add, edit and delete questions in question bank.
\end{itemize}

\subsection{Non Functional Requirements}
\begin{itemize}
    \item NFR1 Performance: Code evaluation result should come in 5 seconds for normal submissions. Pages should load in 2 seconds.
    \item NFR2 Scalability: System should handle 1000 concurrent submissions without major slowdown.
    \item NFR3 Security: Authentication should be token based. Code execution should have strict limits like 256 MB RAM and 1 CPU core.
    \item NFR4 Availability: Platform should maintain 99.5\% uptime.
    \item NFR5 Usability: Student should start a test in 3 clicks without training.
    \item NFR6 Maintainability: System should be modular so one module change should not break others.
\end{itemize}

\section{Architecturally Significant Requirements}

\subsection{ASR1 Secure Code Execution}
Student code is untrusted input. If we run it directly then one bad code can crash full app. So we must run code in sandbox container with CPU, memory and timeout limits.

\subsection{ASR2 Async Submission Processing}
Code compile and run takes time. If API thread waits for it then all other requests become slow. So API should only push job to queue and worker should process in background.

\subsection{ASR3 Clear Modular Boundaries}
Project timeline is short and team size is five students. So modular monolith is practical. We keep auth, test, evaluation and analytics in separate modules with clear boundaries.

\subsection{ASR4 Fast Student Feedback}
Students need result quickly during practice. So queue wait time, worker speed and MongoDB query speed are key architecture points.

\section{Subsystem Overview}

The platform is one modular monolith application with internal modules. Heavy code execution is moved to worker process through queue.

\subsection{Web Application Module}
\begin{itemize}
    \item Web Frontend: Student and admin user interface.
    \item Authentication Module: Registration, login, token creation and access control.
\end{itemize}

\subsection{Test Management Module}
\begin{itemize}
    \item Question Management: Add, update and remove coding, SQL and MCQ questions.
    \item Test Generation: Build timed test by topic, type and difficulty.
\end{itemize}

\subsection{Evaluation Module}
\begin{itemize}
    \item Submission Evaluator: Detect submission type and call correct evaluator.
    \item Code Worker: Pick queued jobs and run code in sandbox container.
    \item SQL Evaluator: Run SQL answer on sample database and compare expected output.
    \item Message Queue: Hold jobs and smooth traffic spike.
\end{itemize}

\subsection{Analytics Module}
\begin{itemize}
    \item Analytics Engine: Calculate score and weak topic trends.
    \item Student Dashboard: Show attempts, scores and progress.
    \item Admin Console: Show usage and performance reports.
\end{itemize}

\subsection{Shared Data Layer}
\begin{itemize}
    \item MongoDB Database: Store users, questions, tests, submissions and analytics data.
\end{itemize}
